⚠ One-time setup required on this device

Services are served over HTTPS using a private local CA. Install the CA cert once and all PicoCluster HTTPS services will be trusted on this device — no more warnings, ever.

Download the CA cert

⬇ Download picocluster-ca.crt
2
Install it on this device

Open Terminal and run:

sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain ~/Downloads/picocluster-ca.crt

Or: open Keychain Access → drag in the file → double-click → set to Always Trust. Restart your browser.

Double-click the downloaded .crt file → Install Certificate → Store Location: Local Machine → Place in: Trusted Root Certification Authorities → Finish. Restart your browser.

Open Safari and tap the download link above. Tap Allow → Close. Go to Settings → General → VPN & Device Management → tap the PicoClaw profile → Install. Then go to Settings → General → About → Certificate Trust Settings → enable PicoClaw Local CA.

Download the cert, then go to Settings → Security → Install from storage (path varies by manufacturer). Select the file, name it PicoClaw CA, and choose VPN & Apps or CA certificate. Chrome requires the cert to be in the user store.

sudo cp ~/Downloads/picocluster-ca.crt /usr/local/share/ca-certificates/picocluster.crt sudo update-ca-certificates

For Chrome/Chromium on Linux: Settings → Privacy → Manage Certificates → Authorities → Import → select the file → check Trust for websites.

Fallback Access — SSH Tunnel

Use the SSH tunnel if mDNS (*.local) doesn't resolve on your network — for example, on some corporate networks or behind certain routers. Primary access is via claw.local and threadweaver.local after installing the CA cert above.

ssh -L 5174:localhost:5174 -L 18790:localhost:18790 picocluster@clusterclaw

Then open:
• https://localhost:5174 — ThreadWeaver (accept self-signed cert warning once)
• https://localhost:18790 — OpenClaw (token: picocluster-token)

Remote Access — Tailscale optional

Tailscale gives the cluster a stable hostname and IP accessible from anywhere — home, office, remote — without port forwarding or VPN configuration. It works through NAT and firewalls. mDNS (claw.local) continues to work on LAN; Tailscale adds remote access on top.

Frequently Asked Questions

Getting started
How do I open ThreadWeaver or OpenClaw?

Step 1 (once per device): Download and install the PicoClaw CA cert from http://clusterclaw.local/ca.crt. See the First-Time Setup section at the top of this page for platform-specific instructions.

Step 2: After installing and restarting your browser:

  • https://threadweaver.local — ThreadWeaver chat
  • https://claw.local — OpenClaw agent dashboard

No SSH tunnel needed. Both are served directly over your LAN via HTTPS.

Why do I need to install a CA cert?

PicoCluster Claw serves all services over HTTPS using TLS certificates signed by its own private Certificate Authority (CA). This gives you real encrypted HTTPS on your local network without depending on an internet-based CA like Let's Encrypt.

Installing the CA cert tells your device to trust certificates issued by PicoCluster Claw. You do this once per device — after that, claw.local and threadweaver.local load like any normal HTTPS site, with no warnings.

The CA cert never leaves your LAN and only signs certs for claw.local and threadweaver.local.

*.local doesn't resolve on my network. What do I do?

*.local names are announced via mDNS (Avahi/Bonjour). Some networks block multicast traffic, preventing mDNS from working. Workarounds:

  • By IP: Use the cluster IP directly — https://10.1.10.220 — but the TLS cert won't match the IP (browser warning). You'll need to add the CA cert and either use the hostname or get a cert with the IP SAN.
  • SSH tunnel: ssh -L 5174:localhost:5174 -L 18790:localhost:18790 picocluster@clusterclaw then https://localhost:5174 and https://localhost:18790.
  • Tailscale: Install Tailscale on clusterclaw and your devices. Get a stable hostname that works anywhere without mDNS.
  • Add to your hosts file: On your computer add 10.1.10.220 claw.local threadweaver.local clusterclaw.local to /etc/hosts (macOS/Linux) or C:\Windows\System32\drivers\etc\hosts (Windows).
Can I access PicoCluster Claw from outside my home network?

Not by default. For remote access, install Tailscale on clusterclaw and your remote devices. Tailscale provides a stable hostname, works through NAT and firewalls, and optionally issues real TLS certificates via tailscale cert — no CA cert install needed on your devices.

ssh picocluster@<tailscale-hostname> then use the same SSH tunnel if you haven't set up Tailscale HTTPS certs.

Models & LLMs
Which model should I use for what?

General chat + tools (default): llama3.1:8b — reliable multi-turn tool use, pre-warmed on boot
Fast one-shots, no tools: llama3.2:3b — 3× faster but degrades on tool chains
Reasoning/math: deepseek-r1:7b — shows its work
Code: qwen2.5:3b or starcoder2:3b
Images: llava:7b or moondream:1.8b

Switch models from the ThreadWeaver settings panel (top right). Switching pre-warms the new model automatically so your first reply after a switch is fast.

Why does llama3.2:3b stop using tools after a few messages?

Small models (3B parameters) handle 1–2 tool calls cleanly but after the third or fourth they start hallucinating tool calls as plain text instead of making real ones. You'll see a ```json block in the message body with a fabricated result that didn't actually happen.

How to spot it: real tool calls show up as 🔧 Tool call: markers. Plain-text JSON without that marker is the model faking it.

Fix: switch to llama3.1:8b in Settings → Local → Model. This is why it's the default.

Can I use Claude, GPT, Gemini, or Grok instead of the local models?

Yes. ThreadWeaver supports five providers: local (Ollama), anthropic, openai, gemini, and grok.

Open ThreadWeaver → Settings → paste your API key → pick a model → chat. You can switch providers mid-conversation. Your API keys are stored encrypted on clusterclaw.

If I use an external LLM, can it still use the local MCP tools?

Yes, fully. Tool execution always happens on clusterclaw. The external LLM only sees tool schemas and decides when to call them. The flow:

  • You ask "make the LEDs purple"
  • ThreadWeaver sends the message + 28 tool schemas to the external API
  • The LLM replies with a tool call
  • ThreadWeaver executes it locally on clusterclaw
  • Result goes back to the LLM, which writes its reply

The external LLM never touches your hardware. In practice, Claude and GPT-4 are often better at multi-step tool workflows than the smaller local models.

What gets sent to external LLM providers?

When you use an external provider: your prompts, the LLM's replies, all 28 tool schemas, tool call results (CPU temp, file contents, model lists), and any attached images.

Not sent: your API keys (used only to authenticate), files outside the sandbox, other conversations, raw local inference traffic.

For strict privacy, stick with the local provider. Everything stays on clusterclaw/clustercrush.

How do I add a new Ollama model?

SSH to clustercrush and use pc-pull:

ssh picocluster@clustercrush
pc-pull llama3.3:70b-instruct-q4_K_M

Browse ollama.com/library for model tags. You can also ask the LLM to do it: "Pull the moondream model from the Ollama library."

Which models support MCP tool calling?

ThreadWeaver auto-detects tool support. If a model doesn't support it, TW shows a notice and falls back to chat-only.

Tool calling: llama3.1:8b, llama3.2:3b, phi3.5:3.8b, qwen2.5:3b, deepseek-r1:7b

Chat-only: gemma3:4b, llava:7b, moondream:1.8b, starcoder2:3b

Tools & capabilities
What can the LLM actually do with the 28 MCP tools?

LEDs (5): colors, animations, progress bars, success/error pulses
System (6): CPU temp, memory, disk, uptime, network, load
Clustercrush (4): list/pull Ollama models, GPU memory, loaded models

Can the LLM read files on my computer?

No. The files MCP server is sandboxed to /tmp/clusterclaw-sandbox inside the ThreadWeaver container. Path traversal attempts fail loudly. Your computer's filesystem is never exposed.

Can I add my own MCP tools?

Yes. Drop a new Python file in /opt/clusterclaw/mcp/, add one line to threadweaver/start.sh, and rebuild: sudo docker compose build threadweaver && sudo docker compose up -d threadweaver. See docs/mcp-tools.md for the ~20-line template.

Operations
How do I shut down or restart the cluster?

From this control panel: use the Cluster Control buttons above.

From the terminal:
ssh picocluster@clusterclaw
pc-restart or pc-shutdown or pc-restart clusterclaw

Blue LED pulses signal the shutdown before the node goes down.

How do I update ThreadWeaver or OpenClaw?

ssh picocluster@clusterclaw, then:

pc-update-threadweaver — rebuild ThreadWeaver from latest main
pc-update-openclaw — rebuild OpenClaw with latest npm package
pc-update — update everything
pc-update-models — update Ollama models on clustercrush

What do the LED colors mean?

Teal scanner: idle, all healthy
Purple pulse: agent is thinking
Cyan flash: agent is using a tool
Green burst: operation succeeded
Red flash: operation failed
Blue progress bar: long operation in progress
Amber solid: warning (high temp, degraded service)
Blue pulse then fade: system shutting down

How do I check if everything is healthy?

The status bar at the top of this page updates every 30 seconds. Run pc-status over SSH for a detailed snapshot: containers, systemd services, HTTP health checks, disk, load, and on clustercrush: Ollama state, loaded models, memory.

A service isn't responding — what do I do?

Run pc-status to identify which service is down.

Container unhealthy: cd /opt/clusterclaw && sudo docker compose logs <service> then sudo docker compose restart <service>

Ollama unreachable: ssh picocluster@clustercrush then sudo systemctl restart ollama

Nuclear option: pc-restart all

Privacy & security
Is my data private when I use the local LLM?

Yes. The local provider runs inference on clustercrush (Jetson Orin Nano) using Ollama. Data path: browser → HTTPS (your LAN) → ThreadWeaver on clusterclaw → Ollama on clustercrush. Nothing leaves your network. Model weights and conversations are stored locally.

What's the difference between ThreadWeaver and OpenClaw?

ThreadWeaver is chat-first. Branching conversations, model switching mid-thread, search, notebooks. Best for talking to the model and using tools interactively.

OpenClaw is an agent framework. Give it a goal, it autonomously plans, uses tools, browses the web, and reports back. Integrates with Telegram/Discord/Slack for mobile access.

Both run simultaneously, share the same Ollama backend, and access the same MCP tools.

How do I change the default passwords and tokens?

SSH password (do this first on each node):

passwd

OpenClaw gateway token:
Edit /opt/clusterclaw/.env → set OPENCLAW_TOKEN=new-token
cd /opt/clusterclaw && sudo docker compose up -d openclaw
