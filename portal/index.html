SSH Tunnel (for ThreadWeaver & OpenClaw)

Both ThreadWeaver and OpenClaw are served over HTTPS via Caddy and are not exposed directly on the LAN. Run this on your computer to tunnel both:

ssh -L 5174:localhost:5174 -L 18790:localhost:18790 picocluster@picoclaw

Then open:
• https://localhost:5174 — ThreadWeaver chat
• https://localhost:18790 — OpenClaw dashboard (token: picocluster-token)

Frequently Asked Questions

Getting started
How do I start chatting with ThreadWeaver?

ThreadWeaver is served over HTTPS through an SSH tunnel. Open a terminal on your computer and run:

ssh -L 5174:localhost:5174 -L 18790:localhost:18790 picocluster@picoclaw

Leave that terminal open. Then in your browser, visit https://localhost:5174. Accept the self-signed cert warning on first visit — the cert is issued by Caddy on picoclaw and your traffic is already encrypted by SSH.

Why do I need an SSH tunnel?

For security. ThreadWeaver and OpenClaw are bound only to picoclaw's loopback interface (127.0.0.1), so nothing on your LAN can reach them directly without authenticating via SSH first.

The tunnel forwards two local ports (5174, 18790) to picoclaw's Caddy HTTPS wrapper, which reverse-proxies to the raw HTTP ports inside picoclaw.

My browser says the certificate is invalid. Is that safe?

Yes, for this setup. Caddy uses a locally-issued certificate for localhost, not one from a public CA (public CAs won't issue for localhost). Your traffic is already encrypted twice: once by SSH (the tunnel) and once by TLS (Caddy). The warning is about trust, not encryption.

Click "Advanced → Proceed to localhost" and you're good.

Can I access PicoClaw from outside my home network?

Not by default. For remote access, install Tailscale on picoclaw and your remote devices. Then use the Tailscale IP instead of the LAN IP in your SSH tunnel command:

ssh -L 5174:localhost:5174 -L 18790:localhost:18790 picocluster@<tailscale-ip>

Tailscale wraps everything in a WireGuard VPN and handles NAT traversal automatically.

Models & LLMs
Which model should I use for what?

General chat: llama3.1:8b — best quality/speed balance
Fast one-shots: llama3.2:3b — starts instantly
Reasoning/math: deepseek-r1:7b — shows its work
Code: qwen2.5:3b or starcoder2:3b
Images: llava:7b or moondream:1.8b
Tool-heavy workflows: llama3.1:8b (most reliable of the local set)

Switch models from the ThreadWeaver settings panel (top right).

Can I use Claude, GPT, Gemini, or Grok instead of the local models?

Yes. ThreadWeaver supports five providers: local (Ollama), anthropic, openai, gemini, and grok.

Open ThreadWeaver → Settings → paste your API key → pick a model → chat. You can switch providers mid-conversation. Your API keys are stored on picoclaw and only used to authenticate with that provider.

If I use an external LLM, can it still use the local MCP tools?

Yes, fully. Tool execution always happens on picoclaw. The external LLM only sees tool schemas and decides when to call them. The flow is:

  • You ask the LLM "make the LEDs purple"
  • ThreadWeaver sends your message plus the 28 tool schemas to the external API
  • The LLM replies with a tool call: {"tool": "set_led_color", "args": {"color": "purple"}}
  • ThreadWeaver executes the tool locally on picoclaw
  • Tool result goes back to the LLM, which writes its final text reply

The external LLM never touches your hardware directly — picoclaw acts as the trusted executor. All 28 tools work with every provider. In practice, external models like Claude and GPT-4 are often better at multi-step tool workflows than the smaller local models.

What gets sent to external LLM providers?

When you use an external provider, the entire conversation flows through that provider's API, including: your prompts, the LLM's replies, all 28 tool schemas, tool call results (CPU temp, file contents, model lists, etc.), and any attached images.

Not sent: your API keys (used only to authenticate), files outside the sandbox, anything on your LAN that the tools don't explicitly fetch, other conversations, raw local model inference.

For strict privacy, stick with the local provider. Everything stays on picoclaw/picocrush.

How do I add a new Ollama model?

SSH to picocrush and use pc-pull:

ssh picocluster@picocrush
pc-pull llama3.3:70b-instruct-q4_K_M

Browse ollama.com/library to find model tags. You can also ask the LLM to do it: "Pull the moondream model from the Ollama library."

Why doesn't the chat respond when I pick gemma3, llava, or moondream?

Some Ollama models don't support tool calling. ThreadWeaver sends the 28 MCP tool schemas on every request, and models that don't support them return a 400 error that looks like "no response" in the UI.

Models that DO support tools: llama3.2:3b (default), llama3.1:8b, phi3.5:3.8b, qwen2.5:3b, deepseek-r1:7b

Models that DON'T support tools: gemma3:4b, llava:7b, moondream:1.8b, starcoder2:3b

If you want to use gemma3 or the vision models, switch the provider to an external LLM (Claude, GPT-4) and keep the tool-capable behavior, or wait for a future ThreadWeaver release that auto-disables tools for non-tool models.

Check sudo docker logs threadweaver --tail 40 | grep -v settings for the actual error if you're not sure.

Tools & capabilities
What can the LLM actually do with the 28 MCP tools?

LEDs (5): colors, animations, progress bars, success/error pulses
System (6): CPU temp, memory, disk, uptime, network, load
Picocrush (4): list/pull Ollama models, GPU memory, loaded models
Time (4): current time, day of week, durations, countdowns
Files (5): sandboxed read/write/list/delete in /tmp/picoclcaw-sandbox
Built-in (4): ThreadWeaver's own file/directory/command tools

Try the Example Prompts section above or see docs/mcp-tools.md for the full reference.

Can the LLM read files on my computer?

No. The files MCP server is sandboxed to /tmp/picoclcaw-sandbox inside the ThreadWeaver container. Python's Path.resolve() + relative_to() checks reject any path that escapes the sandbox, so attempts like ../../../etc/passwd fail loudly.

Your computer's filesystem is never exposed. The LLM talks to tools on picoclaw; picoclaw talks to its own sandbox. There's no path from the LLM to your laptop unless you paste the contents yourself.

Can I add my own MCP tools?

Yes. There's a ~20-line Python template in docs/mcp-tools.md. Drop a new file in /opt/picoclcaw/mcp/, add one line to threadweaver/start.sh, and rebuild the ThreadWeaver container with sudo docker compose build threadweaver && sudo docker compose up -d threadweaver.

ThreadWeaver also supports connecting to any stdio-based MCP server dynamically via its API — no rebuild required for third-party servers.

Operations
How do I shut down or restart the cluster?

From this control panel: use the Cluster Control buttons above.

From the terminal:

ssh picocluster@picoclaw
pc-restart or pc-shutdown
pc-restart picoclaw (one node only)

Blue LED pulses signal the shutdown before the node goes down.

How do I update ThreadWeaver or OpenClaw?

ssh picocluster@picoclaw, then one of:

pc-update-threadweaver — rebuild ThreadWeaver from latest main
pc-update-openclaw — rebuild OpenClaw with latest npm package
pc-update — update everything (containers + portal + models)
pc-update-models — update Ollama models on picocrush

All scripts git pull the PicoClaw repo first. They're safe to run on a live cluster — individual containers restart with brief downtime.

What do the LED colors mean?

Teal scanner: idle, all healthy
Purple pulse: agent is thinking
Cyan flash: agent is using a tool
Green burst: operation succeeded
Red flash: operation failed
Blue progress bar: long operation in progress
Amber solid: warning (high temp, degraded service)
Blue pulse then fade: system shutting down

Drive the LEDs manually via the control panel above, via pc-led on the terminal, or via the LED API at http://picoclaw:7777/.

How do I check if everything is healthy?

The status bar at the top of this page updates every 30 seconds.

For a full snapshot, SSH in and run pc-status on either node. It shows container state, systemd services, HTTP health checks, disk, and load. On picocrush it also shows Ollama state, loaded models, and unified memory usage.

A service isn't responding — what do I do?

First, run pc-status to see which service is actually down.

Container unhealthy: cd /opt/picoclcaw && sudo docker compose logs <service> then sudo docker compose restart <service>

Systemd service: sudo systemctl restart picoclaw-leds or shutdown-api

Ollama unreachable: ssh picocluster@picocrush, then sudo systemctl restart ollama

Nuclear option: pc-restart all

Privacy & security
Is my data private when I use the local LLM?

Yes. The local provider runs inference on picocrush (Jetson Orin Nano) using Ollama. The data path is entirely inside your LAN: browser → SSH tunnel → ThreadWeaver on picoclaw → Ollama on picocrush. Nothing leaves your network.

Model weights are stored locally on picocrush's NVMe. Conversations are stored in ThreadWeaver's database on picoclaw. External API calls only happen if you explicitly switch to an external provider.

What's the difference between ThreadWeaver and OpenClaw?

ThreadWeaver is chat-first. Conversations you can branch, save, search, and switch models in mid-thread. Best for talking to the model and occasionally using tools.

OpenClaw is an agent framework. You give it a goal and it autonomously plans, uses tools, browses the web, and reports back. Also integrates with Telegram/Discord/Slack for mobile access.

Both run simultaneously, both use the same Ollama backend, and both access the same MCP tools.

How do I change the default passwords and tokens?

SSH password (do this first):

passwd — on each node after SSH login

OpenClaw gateway token:

openclaw config set gateway.auth.token "new-token"
sudo docker restart openclaw

Shutdown API token: edit /etc/systemd/system/shutdown-api.service.d/override.conf and set Environment=SHUTDOWN_TOKEN=new-token, then sudo systemctl daemon-reload && sudo systemctl restart shutdown-api.
